# Satisfy prerequisites
Before we start using this code we need to satisfy all the prerequisites described here - [[MLflow on Kubernetes on Linux VMs project - Prerequisites|link]].
# Creating Azure Linux VMs
In order to create VMs using Terraform we need to run the following commands:
```
terraform init # only when running Terraform for the first time in this repository
terraform plan -out main.tfplan
terraform apply main.tfplan
```

In order to destroy all the created resources in Azure we need to run the following commands:
```
terraform plan -destroy -out main.destroy.tfplan
terraform apply main.destroy.tfplan
```
# Connecting to the created VMs from our local computer through SSH
To perform next steps, we will need to connect to VMs using SSH using SSH keys generated by  Terraform (more info - [[MLflow on Kubernetes on Linux VMs project - Generating SSH keys|link]]).

We can connect to the created VMs by using this command on our local computer:
>`ssh username@ip_address`

Here is described how to get values needed for SSH connection:
- **ip_address** - In order to get IP addresses of both created VMs we need to use the Terraform outputs called `public_ip_address_vm_1` and `public_ip_address_vm_2` (more info here - [[MLflow on Kubernetes on Linux VMs project - Terraform resources creation|link]])
- **username**- The username value is the same as the one defined in the `terraform.tfvars` file for the `vm_username` variable (or the default one 'azureadmin' defined in the variables.tf).
# Setting up a Kubernetes cluster
When we run `terraform apply`, Terraform performs the initial setup of Kubernetes ([[Spark Operator on Kubernetes on Linux VMs project - Terraform - VMs setup|link]]) but the worker node is not added to the cluster yet, we need to add it manually by following the steps:
- Connect from our local computer to the VM1 using SSH
- Copy the join command using the `cat /home/azureadmin/k8s_join_workers.txt` command
- Connect to VM2 using SSH
- Run the copied command on the VM2 using `sudo`: `sudo <copied-command>` 
## Verify that the cluster is running
In order to confirm that the cluster is running we can do the following things:
- Run the `kubectl get pods -n kube-system` command
	- It lists required system Pods
	- All the listed Pods there should have status 'Running'
	- Pods might have status 'In progress' at the beginning and change it into 'Running' after some time
- Run the `kubectl get nodes` command
	- It should list two nodes 'master' and 'worker1' with the 'Ready' status. 
	- Again it might take some time before they get from the 'In progress' into 'Ready' status.
	- Before both nodes get status 'Ready', the previous command must show that all the system Pods are Running.
# Restarting CoreDNS
I don't know why but after setting up Kubernetes cluster (joining a worker node) we need to restart coredns deployment. Without that resolving DNS names from inside of a Pod doesn't work. Because of that communication with the MLflow Tracking Server might not work.

We need to run this command from one of the VMs:
>`kubectl -n kube-system rollout restart deployment coredns`

Now check if it has been already restarted and is running:
>`kubectl get pods -n kube-system`

More information about that issue can be found here - [[MLflow on Kubernetes on Linux VMs project - Problems and ideas for improvements|link]], in the "DNS resolution" section.
# Deploying MLflow on Kubernetes
Run this script on the VM1 to create all the resources needed for MLflow ([[MLflow on Kubernetes on Linux VMs project - Deploying the MLflow Tracking Server|link]]):
>`/home/azureadmin/k8s/mlflow_deploy.sh`

Now we can check if all those resources are running:
>`kubectl get pods -n mlflow`

Once all those resources are prepared we can deploy a Pod which will run a Python script for testing connectivity to the Tracking Server using this command:
>`kubectl apply -f /home/azureadmin/k8s/mlflow_test.yaml`

Now we can check progress of our testing Pod and the MLflow Tracking Server pod by executing (after some time they should have the 'Running' status):
>`kubectl get pods -n mlflow`

Once we see that all the Pods are running we can check logs of that Pod to verify that it worked (it might take some time to execute the code):
>`kubectl logs mlflow-test -n mlflow`

If it worked then in those logs we should see should see messages 'Logged a test run' and 'Artifact logged'.

If something doesn't work we can try to:
- Create a testing Pod and from that Pod try to resolve some DNS name:
  >`nslookup google.com`
    
    If that doesn't work, then we need to restart the CoreDNS deployment again.
- Recreate mlflow-test Pod
- Recreate resources created by the mlflow_deploy.sh script
- Recreate the entire VMs (rerun the Terraform code)
